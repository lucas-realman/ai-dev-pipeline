## §1 项目概述与范围

### 1.1 项目定义

**AutoDev Pipeline**（自动化开发流水线）是一个**通用的文档驱动自动化编码平台**。它读取一套标准化需求文档集，由 AI 理解项目全貌后自动拆解为编码任务，通过 SSH 分发到多台开发机器上由 aider (AI 编码 CLI) 执行，然后经过三层自动 Review、自动测试、自动通知报告，形成完整闭环。

**核心价值**：将"人写设计文档 → 机器写代码 → 机器自查 → 机器自测"的全流程自动化。

### 1.2 系统定位

| 维度 | 说明 |
|------|------|
| **类型** | 开发工具链 / 自动化编排平台 |
| **用户** | 独立开发者 / 小团队（1-5人），通过文档驱动 AI 编码 |
| **输入** | 标准化需求文档集（7 类 Markdown + 接口契约 YAML） |
| **输出** | 可运行的代码 + 测试报告 + Release Notes + 钉钉通知 |
| **不做** | IDE 插件、在线代码编辑器、实时协作编辑 |

→ 映射: **SYS-001 §1** (系统上下文边界)

### 1.3 适用范围与边界

**范围内**：
- 文档解析 → 编码任务生成 → SSH 远程 aider 执行 → 三层 Review → pytest 测试 → 钉钉通知
- 动态机器池管理（注册/注销/标签匹配/负载均衡）
- 任务状态机全生命周期管理（CREATED → CLOSED，含反馈闭环）
- Git 自动化（commit/push/pull/tag）

**范围外**：
- 不提供代码编辑 UI（编码由 aider 执行）
- 不管理 vLLM / 大模型训练部署
- 不替代 CI/CD 系统（Jenkins/GitHub Actions），专注开发阶段

### 1.4 术语速查

| 术语 | 定义 | 详见 |
|------|------|------|
| aider | AI 编码 CLI，通过 OpenAI 兼容 API 自动生成代码并提交 | [REF-001](../10-references/REF-001-术语表.md) |
| 文档集 | 一个项目需提供的 7 类 Markdown 文档 + contracts/ | §2, FR-001 |
| 机器池 | 可动态注册/注销的远程开发机器集合 | §2, FR-004 |
| 三层 Review | 静态检查 → 契约对齐 → 设计符合度 | §2, FR-009 |
| 状态机 | 任务从 CREATED → PASSED/ESCALATED 的 11 状态流转 | [OD-002](../04-outline-design/OD-002-数据模型设计.md) |

#### FR-001 详细描述

**标题**: 标准化需求文档集加载

**用户故事**: 作为平台用户，我希望系统能自动加载项目文档集，这样 AI 就能理解整个项目的需求和设计。

**前置条件**:
1. 用户在 `config.yaml` 的 `doc_set` 段中配置了 7 类文档的 glob 路径映射（requirements, system_design, architecture, outline_design, detail_design, test_strategy, contracts）
2. 项目根目录下存在对应的 Markdown 文件

**处理规则**:
1. `DocAnalyzer.load_doc_set()` 按 `config.doc_set` 中定义的 glob 模式逐类扫描文件
2. 每类文档可配置多个 glob 模式（如 `docs/01-requirements/**/*.md`）
3. 文件以 UTF-8 编码读取；读取失败的单个文件记录 WARNING 日志并跳过，不中断整体加载
4. 返回 `Dict[str, str]`，key=文档类型标识，value=文件内容拼接

**输出规格**:
- 成功: 返回包含 ≥1 个有效文档的字典
- 空集: 若全部文档路径匹配失败，记录 ERROR 日志，返回空字典 `{}`

**异常场景**:
| 场景 | 行为 |
|------|------|
| glob 匹配 0 个文件 | 该类型返回空字符串，log.warning |
| 文件编码非 UTF-8 | UnicodeDecodeError → log.warning 跳过 |
| config 未配置 doc_set | 使用内置默认 7 类路径模板 |

**验收标准**：给定一套含 7 类 Markdown 文件的测试文档集，`load_doc_set()` 返回 7 个 key-value 对，每个 value 非空。

#### FR-002 详细描述

**标题**: LLM 驱动的智能任务拆解

**用户故事**: 作为平台用户，我希望系统能读懂需求文档并自动把 Sprint 工作拆分为细粒度编码任务，这样我不需要手动拆 Task。

**前置条件**:
1. FR-001 已成功加载文档集（非空字典）
2. 环境变量 `OPENAI_API_BASE` 和 `OPENAI_API_KEY` 已配置
3. 用户指定了 `sprint_id`（如 Sprint 1）

**处理规则**:
1. `DocAnalyzer.analyze_and_decompose(sprint_id, extra_context?)` 构建分解 Prompt
2. Prompt 包含每类文档（截断至 MAX_DOC_LEN=6000 字符）+ Sprint 上下文 + 额外指令
3. LLM 调用参数: model=config.model (可配置 LLM, 默认值由 Config Schema 定义), temperature=0.2, max_tokens=4096, timeout=180s
4. 要求 LLM 返回 JSON 数组，每个元素为 `{task_id, description, tags, target_dir, acceptance, depends_on, estimated_minutes, context_files}`
5. JSON 解析三级容错: ① 直接 `json.loads` ② 从 ` ```json ``` ` 代码块提取 ③ 定位首个 `[` 到末尾 `]` 子串
6. 每个 task 自动补全缺失字段（默认 tags=[], estimated_minutes=30）

**输出规格**:
- 成功: 返回 `List[CodingTask]`，通常 5~20 个任务
- LLM 故障: log.error 后返回空列表 `[]`（降级为手动拆解模式）

**异常场景**:
| 场景 | 行为 |
|------|------|
| LLM API 未配置 | raise `RuntimeError("LLM API 未配置")` |
| LLM 返回非 JSON | 三级容错解析，全部失败 → `ValueError` 被内部捕获 → 返回 `[]` |
| LLM 超时 (>180s) | httpx.TimeoutException → log.error → 返回 `[]` |
| 文档集为空 | log.error("空文档集") → 返回 `[]` |
| Sprint 不存在于文档中 | LLM 无法匹配内容 → 返回 `[]` 或少量低质任务 |

**验收标准**：给定测试文档集，`analyze_and_decompose(sprint_id=1)` 返回 ≥5 个合法 CodingTask，每个含 task_id + description + tags。

#### FR-003 详细描述

**标题**: 编码任务数据结构规范 (CodingTask)

**用户故事**: 作为调度引擎，我需要每个编码任务包含完整元数据，才能正确调度、匹配机器、判断依赖。

**数据模型 (`CodingTask` dataclass)**:

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `task_id` | `str` | ✅ | 唯一标识，格式如 "TASK-001"、"FIX-003" |
| `description` | `str` | ✅ | 编码任务自然语言描述，作为 aider 指令输入 |
| `tags` | `List[str]` | ✅ | 能力标签，用于机器匹配（如 `["python", "gpu"]`） |
| `context_files` | `List[str]` | ❌ | 任务相关上下文文件路径列表 |
| `depends_on` | `List[str]` | ❌ | 前置依赖任务 ID 列表（拓扑排序依据） |
| `acceptance` | `str` | ❌ | 验收标准文本，传递给 Review 和测试 |
| `estimated_minutes` | `int` | ❌ | 预估耗时，默认 30 分钟 |
| `assigned_machine` | `str` | 自动 | v3 动态分配的机器 ID |
| `target_machine` | `str` | ❌ | v2 兼容的指定目标机器 ID |
| `status` | `TaskStatus` | 自动 | 当前状态 (11 态枚举)，默认 CREATED |
| `retry_count` | `int` | 自动 | 分发重试次数，默认 0 |
| `review_retry` | `int` | 自动 | Review 重试次数，默认 0 |
| `test_retry` | `int` | 自动 | 测试重试次数，默认 0 |
| `fix_instruction` | `str` | 自动 | Review/测试失败时的修复指令 |
| `created_at` | `float` | 自动 | 创建时间戳 |
| `started_at` / `finished_at` | `float` | 自动 | 开始/完成时间戳 |

**计算属性**: `total_retries = review_retry + test_retry`; `effective_machine = assigned_machine or target_machine`

**序列化**: `to_dict()` / `from_dict()` 双向转换，`TaskStatus` 以 `.value` 字符串形式序列化。

**验收标准**: 创建 CodingTask 实例 → `to_dict()` → `from_dict()` 往返序列化后所有字段值不变。

#### FR-004 详细描述

**标题**: 动态机器池管理

**用户故事**: 作为运维人员，我希望能通过配置文件初始化机器池，并在运行时动态增删机器，这样我可以灵活调整算力资源。

**前置条件**:
1. `config.yaml` 包含 `machines` 列表，每台机器定义 machine_id, host, port, user, work_dir, tags 等字段
2. 远程机器 SSH 连通性已验证

**数据模型 (`MachineInfo` dataclass)**:

| 字段 | 类型 | 说明 |
|------|------|------|
| `machine_id` | `str` | 唯一标识 (如 "W1", "gpu-01") |
| `host` | `str` | SSH 主机地址 |
| `port` | `int` | SSH 端口，默认 22 |
| `user` | `str` | SSH 用户名 |
| `work_dir` | `str` | 远程工作目录 |
| `tags` | `List[str]` | 能力标签 (如 `["python", "gpu", "ssd"]`) |
| `aider_prefix` | `str` | aider 命令前缀 |
| `aider_model` | `str` | 该机器使用的 LLM 模型 |
| `status` | `MachineStatus` | 状态枚举: ONLINE / BUSY / OFFLINE / ERROR |
| `current_task` | `str` | 当前执行的任务 ID |
| `load` | `Dict[str, float]` | 负载信息 (cpu_percent, ram_percent, disk_free_gb) |

**处理规则**:
1. `MachineRegistry` 通过 `threading.Lock` 保证所有操作线程安全
2. `load_from_config(machines_config: list)` 批量注册机器
3. `register(MachineInfo)` 注册单台机器（ID 重复时覆盖）
4. `unregister(machine_id) -> bool` 注销机器
5. `get_online_machines()` 返回状态为 ONLINE 的机器列表
6. `get_idle_machines()` 返回 ONLINE 且 current_task 为空的机器
7. `set_busy(machine_id, task_id)` / `set_idle(machine_id)` 状态切换

**验收标准**: 加载含 4 台机器的 config → `get_online_machines()` 返回 4 台 → 注销 1 台 → 返回 3 台 → 线程安全测试无竞态。

#### FR-005 详细描述

**标题**: 基于标签的智能机器匹配与负载均衡

**用户故事**: 作为调度引擎，我需要为每个任务找到能力最匹配且最空闲的机器。

**匹配算法 (三级优先)**:
1. **精确匹配**: 任务 `tags` 完全是机器 `tags` 的子集 → 候选机器集合
2. **降级匹配**: 无精确匹配时，按 tags 交集大小降序排列 → 选交集最多的
3. **兜底匹配**: 任何空闲 ONLINE 机器

**负载均衡**: 候选集中按 `load["cpu_percent"]` 升序排列，选取最低负载的机器。

**异常场景**:
| 场景 | 行为 |
|------|------|
| 无可用机器 | 返回 `None`，任务保持 QUEUED 等待下一批次 |
| 所有机器 BUSY | 返回 `None`（不抢占正在执行的任务） |
| 负载信息缺失 | cpu_percent 默认为 0，不影响匹配 |

**验收标准**: 3 台机器 (tags: ["python"], ["python","gpu"], ["java"]) → 任务 tags=["python","gpu"] → 选中第 2 台。

#### FR-006 详细描述

**标题**: 依赖感知的批次任务调度

**用户故事**: 作为编排引擎，我需要按照任务间的依赖关系，自动计算当前可执行的独立批次，最大化并行效率。

**前置条件**:
1. 任务已入队 (`TaskEngine.enqueue(tasks)`)，每个任务自动创建 `TaskStateMachine` 实例
2. 每个任务的 `depends_on` 字段包含了前置依赖信息

**处理规则**:
1. `TaskEngine.next_batch() -> List[CodingTask]` 实现批次调度:
   - 筛选状态为 QUEUED 的任务
   - 检查每个 QUEUED 任务的 `depends_on` 列表，所有前置任务必须为 PASSED 状态
   - 依赖已满足的任务进入本批次候选
2. 通过 `MachineRegistry.match_machine()` 为每个候选任务匹配可用机器
3. 匹配成功的任务数不超过 `max_concurrent`（默认 4）
4. 未匹配到机器的任务保持 QUEUED，等待下一轮

**存储**: `OrderedDict[str, Tuple[CodingTask, TaskStateMachine]]`，线程安全 (`threading.Lock`)

**异常场景**:
| 场景 | 行为 |
|------|------|
| 循环依赖 | 所有环上任务永远无法满足依赖 → 达到 MAX_ROUNDS(20) 后强制 ESCALATED |
| 依赖任务 FAILED/ESCALATED | 后续依赖任务永久阻塞 → MAX_ROUNDS 后升级 |
| 无可用机器 | 返回空批次 `[]`，主循环等待后重试 |
| 重复 task_id | 后入队的覆盖先入队的（OrderedDict 行为） |

**验收标准**: 3 个任务: A(无依赖) → B(depends A) → C(depends B)，第 1 批返回 [A]，A 完成后第 2 批返回 [B]，B 完成后第 3 批返回 [C]。

#### FR-007 详细描述

**标题**: 智能机器分配与分发

**用户故事**: 作为任务调度器，我需要把每个编码任务精确分配到最合适的开发机器上执行，通过标签匹配确保机器能力与任务需求一致。

**分配策略 (三级)**:
1. 任务指定了 `assigned_machine` → 直接使用该机器（v3 人工指定模式）
2. 任务指定了 `target_machine` → 使用兼容映射（v2 机器代号如 W1→4090）
3. 仅有 `tags` → 调用 `MachineRegistry.match_machine(task.tags)` 动态匹配

**处理规则**:
1. 匹配成功后 `mark_dispatched(task_id)` → 状态 QUEUED → DISPATCHED
2. 机器状态标记为 BUSY，`current_task` 设为该 task_id
3. 主循环通过 `Dispatcher.dispatch_batch(tasks)` 使用 `asyncio.gather` 并行分发

**验收标准**: 含 tags=["python","gpu"] 的任务自动匹配到带 gpu 标签的机器并分发。

#### FR-008 详细描述

**标题**: SSH 远程 aider 编码执行

**用户故事**: 作为分发器，我需要通过 SSH 在远程机器上运行 aider CLI 完成编码任务，并收集执行结果。

**前置条件**:
1. 目标机器 SSH 连通（密钥认证）
2. 目标机器已安装 aider CLI 并配置 LLM API 访问

**处理流程**:
1. `Dispatcher.dispatch_task(task) -> TaskResult` (async):
   a. 构建 Markdown 格式编码指令 (`_build_instruction`)，包含 description + acceptance + fix_instruction
   b. 通过 SCP 将指令文件传输到远程机器临时路径
   c. 构建 bash 执行脚本: `cd work_dir → git reset → aider 执行 → git commit → git push (3次重试)`
   d. 通过 `ssh -T` (远程) 或 `bash -s` (本地) 执行脚本
   e. 解析 stdout 提取变更文件列表 (`Wrote xxx` 或 `create mode xxx`)
   f. 返回 `TaskResult(task_id, exit_code, stdout, changed_files)`

**本地/远程判断**: `_collect_local_ips()` 收集本机 127.0.0.1/hostname/网卡 IP，`_is_local(machine)` 判断本地执行还是 SSH 执行。

**超时与容错**:
| 参数 | 值 | 说明 |
|------|-----|------|
| SSH ConnectTimeout | 10s | 连接超时 |
| ServerAliveInterval | 30s | 心跳保活 |
| SCP 传输超时 | 30s | 指令文件传输 |
| 任务执行超时 | 600s (config.single_task_timeout) | aider 执行超时 |

**异常场景**:
| 场景 | 行为 |
|------|------|
| SSH 连接失败 | TaskResult(exit_code=1, stdout=traceback) |
| 执行超时 (>600s) | asyncio.TimeoutError → exit_code=124 |
| 机器不在 registry | TaskResult(exit_code=1, stdout="机器未找到") |
| aider 返回非 0 退出码 | 如实记录 exit_code 和 stdout，交给后续 Review 判断 |

**验收标准**: 向远程机器分发一个简单编码任务 → aider 执行成功 → 返回 exit_code=0 + 变更文件列表。

### 2.4 自动审查 (AutoReview)

| ID | 需求描述 | 优先级 | 映射 SYS | 映射 MOD | 映射 IF |
|----|---------|--------|---------|---------|--------|
| **FR-009** | Layer 1 静态检查：对生成的 .py 文件执行编译检查和 lint 规则扫描 | P0 | SYS-004 | MOD-007 | IF-007 |
| **FR-010** | Layer 2 契约对齐：调用 LLM 对比代码与 contracts/ 接口签名的一致性 | P0 | SYS-004 | MOD-007 | IF-008 |
| **FR-011** | Layer 3 设计符合度：调用 LLM 从 5 维度评分代码与设计文档的匹配度 | P1 | SYS-004 | MOD-007 | IF-008 |

#### FR-009 详细描述

**标题**: Layer 1 — 静态代码检查

**用户故事**: 作为质量守门人，我需要在 AI 写完代码后立即进行编译和 lint 检查，快速拦截语法错误和严重代码问题。

**前置条件**: 编码任务已完成 (CODING_DONE)，`TaskResult.changed_files` 中有 `.py` 文件

**处理规则**:
1. `AutoReviewer._static_check(files: List[str]) -> ReviewResult`
2. **编译检查**: 对每个 `.py` 文件执行 `py_compile.compile()`，捕获 `SyntaxError`
3. **Lint 检查**: 执行 `ruff check --select E,W,F`，仅检查严重规则类别 (E9/F8/F6/F4)
4. Ruff 执行超时: 30s；ruff 工具不存在时静默跳过 lint 步骤（不失败）
5. 短路逻辑: Layer 1 失败 → 直接返回 `ReviewResult(passed=False)`，不进入 Layer 2/3

**输出**: `ReviewResult(passed, layer="static", issues=[], score=None, fix_instruction="")`

**异常场景**:
| 场景 | 行为 |
|------|------|
| py_compile SyntaxError | passed=False，issues 记录行号和错误信息 |
| ruff 返回非 0 | passed=False，issues 记录各违反条目 |
| ruff 未安装 | log.warning 跳过 lint，仅做编译检查 |
| 无 .py 文件 | 直接通过 (passed=True) |

**验收标准**: 含 `SyntaxError` 的代码 → Layer 1 返回 `passed=False`；合法代码 → `passed=True`。

---

#### FR-010 详细描述

**标题**: Layer 2 — 接口契约对齐检查

**用户故事**: 作为质量守门人，我需要验证 AI 生成的代码是否遵循了 contracts/ 中定义的接口规范，确保模块间接口一致性。

**前置条件**: Layer 1 通过，contracts/ 目录下存在 `.yaml/.yml/.sql` 文件

**处理规则**:
1. `AutoReviewer._contract_check(task, files) -> ReviewResult`
2. 读取 contracts/ 下所有接口定义文件 + 被审查代码文件内容
3. 构建 LLM Prompt: 要求对比代码实现与接口定义的一致性
4. LLM 返回 JSON: `{"passed": bool, "issues": [str], "fix_instruction": str}`
5. LLM 调用参数: temperature=0.1, max_tokens=2048, timeout=120s
6. **contracts 为空**: 跳过 Layer 2，直接进入 Layer 3

**降级策略**: LLM 调用失败 → **降级通过 (score=3.5)**，log.warning 记录原因

**验收标准**: 接口签名不一致的代码（如缺少必填参数）→ Layer 2 返回 `passed=False` + `fix_instruction`。

---

#### FR-011 详细描述

**标题**: Layer 3 — 设计符合度多维度评分

**用户故事**: 作为质量守门人，我需要从多维度评估代码与设计文档的匹配程度，给出量化评分。

**前置条件**: Layer 1 和 Layer 2 均通过

**处理规则**:
1. `AutoReviewer._design_check(task, files) -> ReviewResult`
2. 代码截断至 8000 字符后，与任务 description + acceptance 一起构建 Prompt
3. LLM 从 5 个维度评分 (各 1~5 分):
   - **功能完整性**: 是否实现了任务全部要求
   - **接口正确性**: 函数签名/返回值是否正确
   - **错误处理**: 异常捕获和边界处理是否充分
   - **代码质量**: 风格、可读性、命名规范
   - **可运行性**: 代码是否能直接运行通过
4. 总分 = 5 维度平均分，低于 `config.pass_threshold` (默认 3.5) 则失败
5. 失败时 LLM 同时返回 `fix_instruction` 供重试修复

**降级策略**: LLM 调用失败 → 降级通过 (score=3.5)

**验收标准**: 高质量代码 → 总分 ≥ 3.5 → passed=True；低质量代码 → 总分 < 3.5 → passed=False + fix_instruction。

### 2.6 状态机管理

| ID | 需求描述 | 优先级 | 映射 SYS | 映射 MOD | 映射 IF |
|----|---------|--------|---------|---------|--------|
| **FR-014** | 系统应实现 11 状态任务状态机，校验所有转换的合法性 | P0 | SYS-006 | MOD-009 | IF-010 |
| **FR-015** | 非法状态转换应抛出 StateMachineError，不允许静默跳过 | P0 | SYS-006 | MOD-009 | IF-010 |

#### FR-014 详细描述

**标题**: 11 状态任务生命周期管理

**用户故事**: 作为编排引擎，我需要精确控制每个任务在 11 个状态间的流转，确保不会出现非法跳转或遗漏步骤。

**状态定义** (`TaskStatus` 枚举, 11 态):

```
CREATED → QUEUED → DISPATCHED → CODING_DONE → REVIEW → TESTING → JUDGING → PASSED
                                                                            → FAILED
                                                                  → RETRY
                                                                  → ESCALATED
```

**合法转换表**:

| 源状态 | 允许目标状态 | 触发条件 |
|--------|-----------|---------|
| CREATED | QUEUED | `sm.enqueue()` — 入队 |
| QUEUED | DISPATCHED | `sm.dispatch()` — 分发到机器 |
| DISPATCHED | CODING_DONE | `sm.coding_done(result)` — aider 执行完毕 |
| CODING_DONE | REVIEW | `sm.start_review()` — 进入审查 |
| REVIEW | TESTING | `sm.review_done(result)` — 审查通过 |
| REVIEW | RETRY | `sm.review_done(result)` — 审查失败 + review_retry < max |
| REVIEW | ESCALATED | `sm.review_done(result)` — 审查失败 + 超过重试限 |
| TESTING | JUDGING | `sm.test_done(result)` — 进入评判 |
| JUDGING | PASSED | `sm.judge(result)` — 测试通过 |
| JUDGING | RETRY | `sm.handle_failure()` — 测试失败 + 可重试 |
| JUDGING | ESCALATED | `sm.handle_failure()` — 超限升级人工 |
| RETRY | QUEUED | `sm.requeue()` — 重新入队 |

**重试控制**:
- `max_retries` (默认 3): `total_retries = review_retry + test_retry`
- 超过限制 → ESCALATED（升级人工处理，不再自动重试）

**时间戳记录**: `dispatch()` 设置 `started_at`; `judge()` 通过时设置 `finished_at`

**验收标准**: CREATED → QUEUED → DISPATCHED → ... → PASSED 全链路无异常；CREATED → TESTING 直接跳转 → 抛出 `StateMachineError`。

---

#### FR-015 详细描述

**标题**: 严格状态转换校验

**用户故事**: 作为系统可靠性保障，我需要所有非法状态转换立即被发现并报错，绝不允许静默跳过。

**处理规则**:
1. `TaskStateMachine._transit(new_status)` 在执行前查询 `_TRANSITIONS` 表
2. 若 `new_status` 不在 `_TRANSITIONS[current_status]` 中 → raise `StateMachineError`
3. `StateMachineError` 继承自 `Exception`，包含 `current_status` + `attempted_status` 信息

**验收标准**: 所有 11×11=121 种状态转换组合中，仅合法转换（约 12 种）成功，其余 109 种均抛出 `StateMachineError`。

## §3 非功能需求

> 每项使用 **NFR-xxx** 编号。映射到 SYS/ARCH 层的设计决策。

### 3.1 性能需求 (NFR-001 ~ NFR-003)

| ID | 需求描述 | 指标 | 映射 SYS | 映射 ARCH |
|----|---------|------|---------|----------|
| **NFR-001** | 单个编码任务从分发到 aider 返回结果的端到端延迟 | ≤ 10 分钟（超时 kill） | SYS-003 | ARCH-001 |
| **NFR-002** | 任务调度+分发的调度延迟（不含编码时间） | ≤ 5 秒 | SYS-003 | ARCH-001 |
| **NFR-003** | 每日可完成的独立编码任务数量 | ≥ 50 个（4 台机器并行） | SYS-003 | ARCH-002 |

#### NFR-001 详细描述 — 单任务端到端延迟

**度量方式**: 从 `task.started_at` (dispatch 时刻) 到 `TaskResult` 返回的时间差。

**实现机制**:
- `config.single_task_timeout` 控制 aider 执行超时 (默认 600s = 10min)
- 超时触发 `asyncio.TimeoutError` → 进程被 kill → exit_code=124
- SSH ConnectTimeout=10s + ServerAliveInterval=30s 保障连接级超时

**推导依据**: 基于 4 台异构机器实测，单任务平均 3-5 min，最大 8 min (含 LLM 响应时间)。10 min 上限覆盖 99% 场景。

---

#### NFR-002 详细描述 — 调度延迟

**度量方式**: `next_batch()` 调用耗时 + `match_machine()` 匹配耗时。

**实现分析**:
- `next_batch()` 遍历 OrderedDict，复杂度 O(N)，N=任务总数 (≤200/Sprint)
- `match_machine()` 遍历机器列表，复杂度 O(M)，M=机器数 (≤10)
- 总延迟: O(N×M) ≈ O(2000)，纯内存操作，实测 < 10ms
- 瓶颈在 SCP 传输指令文件 (30s 超时)，但不计入调度延迟

---

#### NFR-003 详细描述 — 日吞吐量

**推导公式**: 日吞吐量 = 机器数 × (工作时间 / 平均任务耗时)
- 4 台机器 × (8h × 60min / 5min/task) = 4 × 96 = 384 tasks/day (理论上限)
- 实际受限于: LLM API 限流、依赖串行化、Review 耗时、网络延迟
- 保守估计 ≥ 50 tasks/day，即机器平均利用率 ≥ 13%

### 3.2 可靠性需求 (NFR-004 ~ NFR-006)

| ID | 需求描述 | 指标 | 映射 SYS | 映射 ARCH |
|----|---------|------|---------|----------|
| **NFR-004** | SSH 连接失败时自动重试 | 重试 3 次，间隔 30s | SYS-003 | ARCH-001 |
| **NFR-005** | Orchestrator 崩溃后自动恢复 | systemd 自动重启，恢复任务队列 | SYS-006 | ARCH-002 |
| **NFR-006** | 任务失败自动重试（Review 失败 + 测试失败合计） | 最多 3 次，超限升级人工 | SYS-004 | ARCH-001 |

#### NFR-004 详细描述 — SSH 连接重试

**实现机制**:
- Dispatcher 中 SSH 命令失败 → TaskResult(exit_code=1)
- TaskStateMachine → RETRY → QUEUED → 重新调度（可能分配到其他机器）
- Git push 在 bash 脚本层面内置 3 次重试 (`for i in 1 2 3; do git push && break; sleep 5; done`)

**当前局限**: SSH 连接级重试在 dispatcher 层未实现指数退避（v2.0 ACTION A-005 已提出）

---

#### NFR-005 详细描述 — 崩溃恢复

**实现机制**:
- 依靠 systemd `Restart=always` + watchdog 保障进程可用性
- **当前局限**: 崩溃恢复后任务队列丢失（纯内存 OrderedDict），重启后需重新发现任务
- **设计改进** (DD-MOD-006 ALG-020~022): 状态持久化到 JSON 文件，恢复时从快照恢复

---

#### NFR-006 详细描述 — 任务级重试

**实现机制**:
- `TaskStateMachine.max_retries=3`，`total_retries = review_retry + test_retry`
- Review 失败: `review_retry++` + `fix_instruction` 传递给下次 aider 执行
- 测试失败: `test_retry++` + 自动生成修复指令
- 超出限制: ESCALATED 状态 → 钉钉通知人工团队

### 3.3 可扩展性需求 (NFR-007 ~ NFR-009)

| ID | 需求描述 | 指标 | 映射 SYS | 映射 ARCH |

**验证方法**: 更换 config.yaml 中的项目路径和文档集路径 → 系统可处理全新项目

## §4 约束与假设

> 每项使用 **CON-xxx** 编号。

### 4.1 技术约束

| ID | 约束描述 | 影响 | 验证方式 |
|----|---------|------|---------|
| **CON-001** | 编码引擎固定为 aider CLI（不自行实现编码能力） | Orchestrator 只做调度，不重复 aider 已有能力 | CI: 确认无直接代码生成逻辑 |
| **CON-002** | LLM 通过 OpenAI 兼容 API 调用（一步 API 网关） | 所有 LLM 调用统一走 `OPENAI_API_BASE` + `OPENAI_API_KEY` | 配置检查 |
| **CON-003** | Python ≥ 3.10，核心依赖：PyYAML, httpx, loguru | 最小化第三方依赖；不使用 asyncssh，基于原生 ssh/scp 命令 | `pyproject.toml` 依赖列表审查 |
| **CON-004** | 配置文件格式为 YAML，环境变量用 `${VAR}` 语法引用 | Config 类递归展开所有 `${VAR}` 引用 | 单元测试覆盖环境变量展开 |

#### CON-001 详细说明 — aider CLI 依赖

**约束原因**: aider 是经过验证的 AI 编码 CLI 工具，已支持 OpenAI/Anthropic/本地模型。重新实现编码能力投入大、无差异化价值。

**影响范围**:
- Dispatcher 通过 bash 脚本调用 `aider --message-file xxx`，不直接调用 LLM 写代码
- 编码质量受 aider 版本和模型能力约束
- aider 升级可能引入 CLI 参数变化风险

**当前使用方式**: `{aider_prefix} aider --model {model} --message-file {msg_path} {context_files}`

---

#### CON-003 详细说明 — 技术栈选型

**选型决策**:
| 依赖 | 用途 | 替代方案评估 |
|------|------|-------------|
| PyYAML | 配置文件解析 | toml/json — YAML 可读性优，支持注释 |
| httpx | LLM API + 钉钉 HTTP | requests — httpx 原生支持 async |
| loguru | 日志 | stdlib logging — loguru 零配置、支持结构化 |
| asyncio | 并发 SSH/测试 | threading — asyncio 适合 I/O 密集场景 |
| ssh/scp 命令行 | 远程执行 | asyncssh 库 — 实测 ssh 命令行更稳定，无额外依赖 |

### 4.2 资源约束

| ID | 约束描述 | 影响 | 缓解措施 |
|----|---------|------|---------|
| **CON-005** | 初始机器池为 4 台异构机器（GPU/SSD/通用） | 并行度上限 = 机器数 | NFR-007 动态扩缩容 |
| **CON-006** | 所有机器位于同一内网段（172.16.x.x），SSH 互通 | 无公网暴露需求，但限制了跨区域部署 | 未来支持 VPN 隧道 |
| **CON-007** | Orchestrator 以单实例运行（无高可用集群） | 单点故障风险 | systemd + watchdog + 状态持久化 (DD-MOD-006) |

| ID | 假设描述 | 风险等级 | 验证状态 |
|----|---------|---------|---------|
| **CON-008** | 每台开发机已预装 aider CLI 并配置好 SSH 密钥 | 低 | ✅ 部署脚本已验证 |
