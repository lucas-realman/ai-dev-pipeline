## 三、设计模式审查

### 3.1 好的设计

| 模式 | 应用点 | 评价 |
|------|--------|------|
| **状态模式 (State Pattern)** | TaskStateMachine 11 态 + _TRANSITIONS 表驱动 | 优秀 — 表驱动避免 if/else 膨胀 |
| **策略模式 (Strategy)** | Dispatcher 本地 vs SSH 自动选择 | 好 — 利于扩展到 K8s 等 |
| **管道过滤器 (Pipeline)** | 主循环: 解析→调度→编码→审查→测试→判定 | 优秀 — 每阶段独立可替换 |
| **短路评估** | Review L1→L2→L3 失败即停 | 优秀 — 节省 API 成本 |
| **组合根 (Composition Root)** | main.py 统一组装所有模块 | 好 — 便于依赖注入 |

### 3.2 需要改进的设计

| # | 问题 | 当前做法 | 建议 |
|---|------|---------|------|
| 1 | **缺少接口抽象** | dispatcher/reviewer/test_runner 直接使用具体类 | 定义 Protocol/ABC 接口，支持 mock 和替换 |
| 2 | **日志无结构化** | OD-003 §3 提到 `[module] level message` 格式 | 建议采用 structlog 或至少 JSON 格式化，便于 grep/分析 |
| 3 | **错误处理不统一** | StateMachineError、httpx.HTTPError、SSH 超时各自处理 | 建议定义 `PipelineError` 基类 + 错误码枚举 |
| 4 | **CodingTask 字段过多** | 18+ 字段的 dataclass | 考虑拆分为 `TaskSpec` (不变部分) + `TaskState` (动态部分) |
| 5 | **缺少幂等设计** | dispatch_task 如果中途失败，重新分发可能在远程机器留下脏状态 | 添加 cleanup 机制或分发前检查 |

## 四、性能与可扩展性评估

### 4.1 性能瓶颈分析

| 环节 | 预期耗时 | 瓶颈分析 |
|------|---------|---------|
| 文档拆解 (LLM) | 10~30s | 受 LLM API 响应时间制约，无缓存 |
| 任务排队+调度 | <1s | 内存操作，不是瓶颈 |
| SSH 分发 | 2~5s/task | asyncssh 并行已优化 |
| aider 编码 | 3~15min/task | 瓶颈在 LLM 推理，非系统层面 |
| 三层 Review | 30~90s | L2+L3 各一次 LLM 调用 |
| pytest 测试 | 5~30s | 取决于项目测试量 |

**结论**: 系统本身不是瓶颈，主要等待时间在 LLM API 响应。4 台并行机器已最大化 LLM API 的利用率。

### 4.2 扩展性评估

| 场景 | 支持度 | 改造成本 |
|------|--------|---------|
| 机器池从 5→10 | ✅ YAML 配置 | 零代码变更 |
| 机器池 >20 | ⚠️ async 可能不够 | 需引入 Redis 队列 (ADR-005 已预见) |
| 多项目并发 | ⚠️ 单 orchestrator 进程 | 需实例隔离或多租户改造 |
| LLM 供应商切换 | ⚠️ 多处直调 | 需抽取 LLMClient (见 §三) |
