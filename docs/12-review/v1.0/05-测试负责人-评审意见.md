## 三、测试自动化评审

### 3.1 自动化可行性

| 层 | 自动化难度 | 当前状态 | 建议 |
|----|-----------|---------|------|
| L1 冒烟 | 低 | conftest.py + markers 已规划 | ✅ Sprint 0 即可实现 |
| L2 组件 | 中 | 需要 mock LLM/SSH/钉钉 | 需要建立统一 mock 框架 |
| L3 集成 | 高 | 跨模块 + 外部服务 | 需要 docker-compose 测试环境 |
| L4 验收 | 不适合自动化 | 含人工判断 | 保持人工执行 |

### 3.2 🟡 缺失: Mock 框架规划

TEST-001 §3.4 提到 `tests/conftest.py` 共享 fixture，但缺少:
- 统一的 LLM mock (返回预设 JSON)
- SSH mock (模拟 asyncssh 连接)
- 钉钉 mock (mock httpx.post)
- 文件系统 mock (临时项目目录)

**建议**: 创建 `tests/conftest.py` 的详细设计，列出所有 mock fixture。

### 3.3 🟡 缺失: 测试数据管理

TEST-001 §3.4 对测试数据描述过于简略:
- 无示例 config.yaml (测试用)
- 无示例文档集 (doc_set fixture)
- 无预期 LLM JSON 响应样本
