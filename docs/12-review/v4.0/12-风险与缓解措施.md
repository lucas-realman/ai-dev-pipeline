# 风险与缓解措施评审

> **评审版本**: v4.0  
> **评审日期**: 2026-03-07  
> **评审范围**: 文档体系暴露的技术风险、实现风险、运维风险及缓解策略

---

## 风险分级标准

| 级别 | 定义 | 响应时间 |
|------|------|---------|
| 🔴 P0 Critical | 阻塞系统实现, 接口断裂或数据丢失 | 立即修复 |
| 🟡 P1 High | 功能缺失或行为不可预测, 不阻塞但影响质量 | 1-2 天内修复 |
| 🔵 P2 Medium | 文档不一致或引用错误, 影响维护效率 | 1 周内修复 |
| 🟢 P3 Low | 格式/命名/编号问题, 不影响功能 | 下一迭代修复 |

## 🔴 P0 — Critical 风险

### RISK-P0-001: Orchestrator↔TaskEngine 接口断裂

| 项目 | 详情 |
|------|------|
| **风险描述** | Orchestrator (DD-MOD-013) 调用 TaskEngine 的 7 个方法名与 TaskEngine (DD-MOD-004) 定义不匹配。实现后集成测试将全面失败 |
| **影响** | 系统核心流程无法运行 |
| **概率** | 100% (确定发生) |
| **缓解措施** | 以 DD-MOD-004 为权威来源, 统一 DD-MOD-013 ALG-031 中所有方法调用名 |
| **工作量** | 1h 文档修改 |
| **v3.0 关联** | F-001, ⬜ 未修复 |

### RISK-P0-002: StateMachine 非法转移

| 项目 | 详情 |
|------|------|
| **风险描述** | ALG-009 从 CREATED 调用 `sm.escalate()`, 但转移表 `_TRANSITIONS[CREATED]=[QUEUED]` 不包含 ESCALATED |
| **影响** | 异常任务无法升级, 系统卡死 |
| **概率** | 100% |
| **缓解措施** | 方案A: 在转移表中增加 `CREATED→ESCALATED`; 方案B: 修改 ALG-009 先 `CREATED→QUEUED` 再 `QUEUED→ESCALATED` |
| **推荐** | 方案A — 更简洁 |
| **工作量** | 0.5h |
| **v3.0 关联** | F-002, ⬜ 未修复 |

### RISK-P0-003: ALG 编号全局冲突

| 项目 | 详情 |
|------|------|
| **风险描述** | DD-MOD-001 ALG-005/006 与 DD-MOD-002 ALG-005/006 定义完全不同的算法, DD-001 索引只映射到 DD-MOD-002 版本 |
| **影响** | 实现者无法确定 ALG-005 到底实现哪个算法 |
| **概率** | 100% |
| **缓解措施** | DD-MOD-001 的两个算法重新编号为 ALG-032/033, 更新 DD-001 索引 |
| **工作量** | 1h |
| **v3.0 关联** | 新发现 |

### RISK-P0-004: 构造函数签名断裂

| 项目 | 详情 |
|------|------|
| **风险描述** | DD-MOD-001 DocAnalyzer `__init__(project_path, ...)` vs DD-MOD-013 `DocAnalyzer(config)` — 参数完全不匹配 |
| **影响** | Orchestrator 初始化阶段崩溃 |
| **概率** | 100% |
| **缓解措施** | 统一所有模块构造函数为 `__init__(config: Config)`, 由 Config 提供所需参数 |
| **工作量** | 1.5h (涉及 DD-MOD-001/002/013) |
| **v3.0 关联** | 新发现 |
