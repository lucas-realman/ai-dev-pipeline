# v2.1 接口可行性评审

> **评审编号**: REVIEW-v2.1-02  
> **评审范围**: 12 个内部接口 (IF-001~012) + 3 个外部接口的可行性  
> **评审日期**: 2026-03-07  
> **评审依据**: OD-003-接口契约设计, DD-MOD-003, 源码实现  
> **评审结论**: ⚠️ 有条件可行 — 核心接口已实现，契约文件缺失，3 处接口需增强

---

## §1 评审方法

对每个接口按以下维度评审:

| 维度 | 说明 |
|------|------|
| **签名一致性** | 设计文档接口签名 vs 实际代码是否一致 |
| **参数完整性** | 必要参数是否齐全，类型提示是否完整 |
| **返回值可靠性** | 返回值类型是否明确，异常路径是否有返回 |
| **错误处理覆盖率** | 文档声明的异常是否在代码中实际捕获处理 |
| **契约文件状态** | contracts/ 下是否有对应的 YAML 定义 |

|------|------|
| **设计签名** | `async def dispatch_task(task: CodingTask) -> TaskResult` |

### EXT-001: LLM API (OpenAI 兼容)

| 维度 | 评估 |
|------|------|
| **端点** | `POST {OPENAI_API_BASE}/chat/completions` |
| **实际调用** | doc_analyzer._call_llm() + reviewer._call_llm() — 完全一致 ✅ |
| **认证** | `Authorization: Bearer {OPENAI_API_KEY}` ✅ |
| **超时** | doc_analyzer: 120s, reviewer: 120s ✅ |
| **重试** | ❌ 两处 _call_llm 均无重试机制 |
| **模型参数** | doc_analyzer: temperature=0.2, reviewer: temperature=0.1 ✅ (差异合理) |
| **响应解析** | 3 级 JSON fallback (直接解析→代码块正则→首尾花括号) ✅ |

**风险评估**:

**结论**: ⚠️ 有条件可行 — 需增加 429 限流重试和指数退避

### EXT-003: SSH 远程执行协议

| 维度 | 评估 |
|------|------|
| **协议** | OpenSSH 兼容，密钥认证 (ed25519) |
| **SCP** | `scp -q -o ConnectTimeout=10 -P {port}` ✅ |
| **SSH exec** | `ssh -T -p {port} -o ConnectTimeout=10 -o ServerAliveInterval=30` ✅ |
| **超时层** | L1: ConnectTimeout=10s, L2: ServerAliveInterval=30s, L3: asyncio.wait_for(timeout) ✅ |
| **本地检测** | `_is_local()` 比对 hostname/IP → 本地执行跳过 SSH ✅ |
| **脚本注入** | 通过 stdin pipe 发送脚本，避免命令行参数注入 ✅ |
| **并发推送** | 3 次重试 (rebase → merge → sleep) 处理 git push 冲突 ✅ |

**风险评估**:
- SSH 密钥未部署: SCP 失败 → dispatch_task 返回 exit_code=1 ✅
- 远端 aider 未安装: 脚本执行失败 → stderr 包含错误信息 ✅
- 网络超时: 3 层超时机制保障 ✅

**结论**: ✅ 接口可行 — 最成熟的外部接口，已在多机环境实战验证

## §4 契约文件差距分析

| 项目 | 设计期望 | 实际状态 | 差距 |
|------|---------|---------|------|
| `contracts/` 目录 | 存放 YAML/SQL 接口契约 | ❌ 目录为空 | 无契约文件 |
| Reviewer L2 | 读取 contracts/ 做契约对齐检查 | 无契约时 auto-pass (score=5.0) | 功能退化 |
| Dispatcher | `--read contracts/{f.name}` 传给 aider | 无文件可读 | aider 无契约约束 |

**影响分析**:
- contracts/ 为空导致 Reviewer Layer 2 始终 auto-pass — **审查深度降低**
- aider 缺少契约引导 — **生成代码质量依赖 Prompt 而非形式化接口**
- 这是**已知的技术债务**，不影响可行性但影响质量
