# 安全工程师评审意见

> **评审版本**: v4.0  
> **评审日期**: 2026-03-07  
> **评审范围**: AI-Dev-Pipeline 全量文档体系 — 安全设计与威胁建模

---

## 总体评价

系统的安全设计在文档层面有良好基础: 沙箱隔离 (DD-MOD-007 §3a)、路径遍历防护、敏感信息脱敏 (API Key → `***`)、CON-003 禁止 asyncssh (使用 `subprocess+ssh`)。对于一个管理远程机器代码部署的系统，这些安全控制是必要的。

但存在 **文档描述矛盾** (OPS-003 RISK-002 提及 asyncssh) 和 **安全边界未完全定义** 的问题。

## 亮点

| # | 安全控制 | 评价 |
|---|---------|------|
| 1 | **SSH via subprocess** (非 asyncssh) | 利用系统级 SSH 安全配置, 减少第三方库攻击面 |
| 2 | **沙箱隔离** (DD-MOD-007 §3a) | 远程编码在隔离环境执行, 防止代码逃逸 |
| 3 | **路径遍历防护** | 文件操作限制在 project_path 内 |
| 4 | **API Key 脱敏** | 日志和报告中敏感信息替换为 `***` |
| 5 | **ERR-012~014 SSH 异常体系** | SSH 连接/命令/超时异常分类处理, 安全相关错误可审计 |

## 问题与风险

### 🟡 P1 — 文档矛盾

| # | 问题 | 安全影响 | 关联文档 |
|---|------|---------|---------|
| 1 | **OPS-003 RISK-002 提及 asyncssh** | 如果实现者参照 OPS-003 引入 asyncssh, 将增加第三方依赖攻击面, 且与 CON-003 矛盾 | OPS-003, CONTRACT-003 |
| 2 | **LLM API 密钥传递方式不一致** | DD-MOD-001 构造函数接收 `llm_key` 明文参数; DD-MOD-012 Config 中存储密钥 — 密钥暴露在内存中的方式需统一 | DD-MOD-001, DD-MOD-012 |

### 🔵 P2 — 安全边界待定义

| # | 问题 | 建议 |
|---|------|------|
| 3 | **SSH 密钥管理**未明确描述 — 密钥存储位置、轮换策略、权限控制 | 在 OPS-001 Runbook 中增加 SSH 密钥管理章节 |
| 4 | **LLM API 响应注入风险**未评估 — 恶意 LLM 响应可能包含代码注入 | 在 DD-MOD-008 AutoReviewer 中增加 LLM 响应清洗步骤 |
| 5 | **远程机器访问控制**粒度未定义 — 哪些用户/角色可以触发部署 | 在 REQ-001 中增加访问控制需求 |
| 6 | **日志中的敏感代码**——远程编码日志可能包含业务代码, 存储和访问控制未定义 | 在 OPS-001 中增加日志访问控制 |

### 🟢 P3 — 增强建议

| # | 建议 |
|---|------|
| 7 | 增加安全审计日志 (who/when/what 操作记录) |
| 8 | 定义 LLM Provider 白名单机制 (防止连接到恶意 LLM 服务) |
| 9 | 增加网络隔离说明 (SSH 通信是否走 VPN / 专线) |

## 具体建议

### 建议 1: 消除 asyncssh 矛盾
OPS-003 RISK-002 应删除 asyncssh 引用, 改为描述 `subprocess.create_subprocess_exec` + SSH 的风险和缓解。

### 建议 2: 统一密钥管理方式
推荐: Config 从环境变量或 Secret Manager 加载密钥, 构造函数不接收明文密钥参数。DD-MOD-001/DD-MOD-012 需同步更新。

### 建议 3: 补充 LLM 响应安全清洗
在 DD-MOD-008 ALG-015 (评审主流程) 中增加 LLM 响应格式校验步骤, 防止格式异常或注入内容。

## 结论

- **评审结论**: ⚠️ 有条件通过
- **安全评分**: 3.5/5 — 基础安全控制到位, 但文档矛盾和边界定义需补强
- **优先修复**: P1 #1 (asyncssh 矛盾) + P1 #2 (密钥传递统一)
