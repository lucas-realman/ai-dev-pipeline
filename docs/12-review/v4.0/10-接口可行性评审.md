# 接口可行性评审

> **评审版本**: v4.0  
> **评审日期**: 2026-03-07  
> **评审范围**: 12 个内部接口 (IF-001~012) + 3 个外部 API + 3 份 YAML 契约

---

## 评审方法

对每个接口检验:
1. **参数一致性**: 调用方传参 = 定义方签名
2. **返回值一致性**: 调用方期望 = 定义方返回
3. **异常契约**: 调用方 catch = 定义方 raise
4. **契约对齐**: YAML 契约 = 详细设计

---

## 接口逐一评估

### IF-001 DocAnalyzer → LLMProvider ⚠️

| 检查项 | 结果 |
|--------|------|
| 参数 | `call(prompt: str)` vs `call(messages: List[Dict])` — **不一致** (P2) |
| 返回值 | `str` — 一致 |
| 异常 | `LLMTimeoutError`, `LLMRateLimitError` — 一致 |
| **问题** | CONTRACT-001 定义 `call(messages: List[Dict])`, DD-SYS-001 定义 `call(prompt: str)` |

### IF-002 DocParser → TaskEngine ✅

| 检查项 | 结果 |
|--------|------|
| 参数 | `add_tasks(tasks: List[CodingTask])` — 一致 |
| 返回值 | `None` — 一致 |
| 异常 | `TaskValidationError` — 一致 |

### IF-003 TaskEngine → StateMachine ⚠️ P0

| 检查项 | 结果 |
|--------|------|
| 参数 | `transition(task_id, event)` — 一致 |
| **问题** | ALG-009 调用 `sm.escalate()` from CREATED, 但 `_TRANSITIONS[CREATED]=[QUEUED]` 不包含 ESCALATED — **非法转移** |

### IF-004 TaskEngine → Dispatcher ✅

| 检查项 | 结果 |
|--------|------|
| 参数 | `dispatch(task: CodingTask, machine: MachineInfo)` — 一致 |
| 返回值 | `DispatchResult` — 一致 |
| 异常 | `SSHConnectionError`, `SSHCommandError` — 一致 |

### IF-005 Dispatcher → MachineRegistry ⚠️ P1

| 检查项 | 结果 |
|--------|------|
| 参数 | `get_available()`, `mark_busy()`, `mark_idle()` — 一致 |
| **问题** | Orchestrator 调用 `get_busy_machines()` 和 `set_offline()` — **MachineRegistry 未定义这两个方法** |

### IF-006 Dispatcher → GitOps ✅

| 检查项 | 结果 |
|--------|------|
| 参数 | `create_branch()`, `push()`, `merge()` — 一致 |
| 返回值 | `GitResult` — 一致 |
| 异常 | `GitConflictError`, `GitPushError` — 一致 |

### IF-007 AutoReviewer → LLMProvider ⚠️ P1

| 检查项 | 结果 |
|--------|------|
| **问题** | AutoReviewer 使用 `aiohttp.ClientSession` 直接调用 LLM API, 而非通过 LLMProvider 接口 — 绕过了 Token Bucket 限速 |
| 参数 | 不走 IF-007, 直接 HTTP — **接口旁路** |

### IF-008 TestRunner → Dispatcher ✅

| 检查项 | 结果 |
|--------|------|
| 参数 | SSH 执行 `pytest` 命令 — 通过 Dispatcher 的 SSH 通道 |
| 返回值 | 标准输出解析 — 一致 |

### IF-009 Reporter → TaskEngine ✅

| 检查项 | 结果 |
|--------|------|
| 参数 | `get_all_tasks()` — 一致 (注: Orchestrator 侧写 `get_all_results()`, 但 Reporter 直接调 TaskEngine 用对了方法名) |

### IF-010 StateMachine (内部) ✅

| 检查项 | 结果 |
|--------|------|
| 状态转移表 | 7态13转移, 定义清晰 |
| **问题** | CREATED→ESCALATED 已在 IF-003 记录 |

### IF-011 Reporter → Orchestrator ⚠️ P1

| 检查项 | 结果 |
|--------|------|
| **问题** | Orchestrator 调用 `reporter.notify_shutdown()` 但 Reporter 未定义此方法 |
| **TRACE 问题** | TRACE-001 FR-016/017/018 映射到 IF-010, 应为 IF-011 |

### IF-012 Config → All Modules ⚠️ P2

| 检查项 | 结果 |
|--------|------|
| **问题** | Config Schema 缺少 `per_machine_branch` 字段 (DD-MOD-007 引用) |
| 参数注入 | DD-MOD-001/002 构造函数与 Config 注入方式不统一 |
