## §1 评审概览

### 1.1 评审背景

经过三轮评审 (v1.0 → v2.0 → v2.1) 和两批文档修订 (P0+P1 → P2)，文档体系已覆盖全部模块的详细设计、算法伪代码、序列图、异常层次、日志规范、监控指标、持久化和沙箱方案。

本轮 v3.0 评审聚焦于**前三轮遗留的跨文档一致性问题**——即：一个模块的接口定义是否与调用方的描述完全匹配、变更 (★v1.1/★v1.2) 是否传播到所有引用位置。

> 这些问题会导致按照文档实现时代码无法编译/运行。

| # | 问题 | 定义侧 | 调用侧 | 影响 |

## §4 问题归因分析

### 4.1 接口断裂根因

```
原因 1: DD-MOD 各文件从 DD-001 独立拆分时，方法命名未做统一对账
  └─ DD-MOD-004 用 handle_xxx_done() 风格
  └─ DD-MOD-013 用 handle_xxx_passed/failed() 风格
  └─ 各自独立、且无交叉校验

原因 2: v1.1/v1.2 增量修改时新增方法/字段未同步到被依赖模块
  └─ DD-MOD-013 新增 ALG-030b (Stale-busy) 引用 machine.busy_since
  └─ 但 DD-MOD-005 MachineInfo 字段列表未同步更新

原因 3: 契约 YAML 编写时对 ALG 编号的引用基于记忆，未逐一核对源文档
  └─ CONTRACT-002 写 ALG-012 (实际 ALG-013a)
  └─ CONTRACT-003 写 ALG-010 (实际 ALG-015)
```

### 4.2 变更传播不彻底根因

```
降级 score 修改:
  已传播: DD-MOD-008 §2.4, DD-SYS-001 §2.4
  未传播: REQ-001 (2处), DD-SYS-001 SEQ-SYS-003, OD-MOD-007 (4处)
  
根因: 修改时仅更新了「设计决策描述段落」，未全文搜索 "score=4.0" 做传播
```

## §5 修复建议与优先级

### 5.1 修复路线图

```
Batch 1 — P0 接口断裂 (估计 2h，阻塞实现):
  ├─ F-001: DD-MOD-013 ALG-031 方法名对齐到 DD-MOD-004
  └─ F-002: DD-MOD-006 _TRANSITIONS 增加 CREATED→ESCALATED

Batch 2 — P1 数据模型 + 传播 (估计 3h):
  ├─ F-003: DD-MOD-005 CodingTask 增加 module_name
  ├─ F-004: DD-MOD-005 MachineInfo 增加 busy_since
  ├─ F-005: DD-MOD-005 TestResult 增加 pass_rate/reason/skipped_count
  ├─ F-006: 全文替换 score=4.0→3.5 (REQ-001, DD-SYS-001 SEQ-SYS-003, OD-MOD-007)
  ├─ F-007: DD-MOD-003 增加 get_busy_machines()/set_offline()
  └─ F-008: DD-MOD-010 增加 notify_shutdown()

Batch 3 — P2 跨模块引用 + 格式 (估计 2h):
  ├─ F-009~F-016: 枚举名/字段名/ALG 编号对齐
  └─ F-017~F-021: 章节编号修正

Batch 4 — P3 文档质量 (估计 1h):
  └─ F-022~F-029: 版本号/导航/工时/编辑痕迹清理
```

### 5.2 修复原则

| 原则 | 规则 |
|------|------|
| **定义侧为权威** | 方法签名以被调用模块的定义为准；调用方文档对齐 |
| **dataclass 字段以 DD-MOD-005 为权威** | 其他模块引用的字段必须在 DD-MOD-005 中有定义 |
| **全文搜索传播** | 每次修改值型变更 (如 score 阈值) 时，grep 全文档确认传播 |
| **契约 YAML 对齐详细设计 ALG 编号** | 契约中的 reference 字段必须与 DD-MOD 实际 ALG 编号一致 |
