## §1 项目概述与范围

### 1.1 项目定义

**AutoDev Pipeline**（自动化开发流水线）是一个**通用的文档驱动自动化编码平台**。它读取一套标准化需求文档集，由 AI 理解项目全貌后自动拆解为编码任务，通过 SSH 分发到多台开发机器上由 aider (AI 编码 CLI) 执行，然后经过三层自动 Review、自动测试、自动通知报告，形成完整闭环。

**核心价值**：将"人写设计文档 → 机器写代码 → 机器自查 → 机器自测"的全流程自动化。

### 1.2 系统定位

| 维度 | 说明 |
|------|------|
| **类型** | 开发工具链 / 自动化编排平台 |
| **用户** | 独立开发者 / 小团队（1-5人），通过文档驱动 AI 编码 |
| **输入** | 标准化需求文档集（7 类 Markdown + 接口契约 YAML） |
| **输出** | 可运行的代码 + 测试报告 + Release Notes + 钉钉通知 |
| **不做** | IDE 插件、在线代码编辑器、实时协作编辑 |

→ 映射: **SYS-001 §1** (系统上下文边界)

### 1.3 适用范围与边界

**范围内**：
- 文档解析 → 编码任务生成 → SSH 远程 aider 执行 → 三层 Review → pytest 测试 → 钉钉通知
- 动态机器池管理（注册/注销/标签匹配/负载均衡）
- 任务状态机全生命周期管理（CREATED → CLOSED，含反馈闭环）
- Git 自动化（commit/push/pull/tag）

**范围外**：
- 不提供代码编辑 UI（编码由 aider 执行）
- 不管理 vLLM / 大模型训练部署
- 不替代 CI/CD 系统（Jenkins/GitHub Actions），专注开发阶段

### 1.4 术语速查

| 术语 | 定义 | 详见 |
|------|------|------|
| aider | AI 编码 CLI，通过 OpenAI 兼容 API 自动生成代码并提交 | [REF-001](../10-references/REF-001-术语表.md) |
| 文档集 | 一个项目需提供的 7 类 Markdown 文档 + contracts/ | §2, FR-001 |
| 机器池 | 可动态注册/注销的远程开发机器集合 | §2, FR-004 |
| 三层 Review | 静态检查 → 契约对齐 → 设计符合度 | §2, FR-009 |
| 状态机 | 任务从 CREATED → PASSED/ESCALATED 的 11 状态流转 | [OD-002](../04-outline-design/OD-002-数据模型设计.md) |

## §2 功能需求清单

> 每项功能需求使用 **FR-xxx** 编号，并标注向下映射的 SYS/MOD/IF ID。
> 优先级: **P0**=必须 **P1**=重要 **P2**=可选

### 2.1 文档解析与任务拆解

| ID | 需求描述 | 优先级 | 映射 SYS | 映射 MOD | 映射 IF |
|----|---------|--------|---------|---------|--------|
| **FR-001** | 系统应支持加载标准化需求文档集（7 类 Markdown + contracts/），按 glob 模式匹配 | P0 | SYS-001 | MOD-001 | IF-001 |
| **FR-002** | 系统应调用 LLM 理解文档全貌，自动将指定 Sprint 的工作拆解为独立编码任务（CodingTask） | P0 | SYS-001 | MOD-001 | IF-002 |
| **FR-003** | 每个生成的编码任务应包含：task_id, description, context_files, depends_on, acceptance, tags, estimated_minutes | P0 | — | MOD-001 | IF-002 |

**验收标准**：给定一套测试文档集，`DocAnalyzer.analyze_and_decompose(sprint_id=1)` 返回 ≥5 个合法 CodingTask。

### 2.2 机器池管理

| ID | 需求描述 | 优先级 | 映射 SYS | 映射 MOD | 映射 IF |
|----|---------|--------|---------|---------|--------|
| **FR-004** | 系统应支持从 config.yaml 加载初始机器列表，每台机器包含 machine_id, host, port, user, work_dir, tags | P0 | SYS-002 | MOD-003 | IF-005 |
| **FR-005** | 系统应支持运行时注册/注销机器（未来通过 REST API） | P1 | SYS-002 | MOD-003 | IF-005 |

**验收标准**：加载含 4 台机器的 config.yaml 后，`registry.get_online_machines()` 返回 4 台；注销 1 台后返回 3 台。

### 2.3 任务调度与 SSH 执行

| ID | 需求描述 | 优先级 | 映射 SYS | 映射 MOD | 映射 IF |
|----|---------|--------|---------|---------|--------|
| **FR-006** | 系统应维护任务队列，按拓扑排序处理依赖关系 | P0 | SYS-003 | MOD-004 | IF-003 |
| **FR-007** | 系统应根据任务 tags 与机器 tags 进行能力匹配，选择最空闲的匹配机器 | P0 | SYS-003 | MOD-003, MOD-004 | IF-004, IF-005 |
| **FR-008** | 系统应通过 SSH 连接远程机器，构建 aider CLI 命令并执行，收集 exit_code 和 stdout | P0 | SYS-003 | MOD-006 | IF-006, IF-007 |

**验收标准**：3 个有依赖关系的任务入队后，系统按依赖顺序分发到可用机器执行。

### 2.4 自动审查 (AutoReview)

| ID | 需求描述 | 优先级 | 映射 SYS | 映射 MOD | 映射 IF |
|----|---------|--------|---------|---------|--------|
| **FR-009** | Layer 1 静态检查：对生成的 .py 文件执行 `py_compile` + `ruff` + `mypy` | P0 | SYS-004 | MOD-007 | IF-007 |
| **FR-010** | Layer 2 契约对齐：调用 LLM 对比代码与 contracts/*.yaml 的接口签名一致性 | P0 | SYS-004 | MOD-007 | IF-008 |
| **FR-011** | Layer 3 设计符合度：调用 LLM 对比代码与设计文档，输出 5 维度评分（1-5 分） | P1 | SYS-004 | MOD-007 | IF-008 |

**验收标准**：含语法错误的代码 → Layer 1 返回 `passed=False`；接口签名不一致的代码 → Layer 2 返回不通过。

### 2.5 测试运行

| ID | 需求描述 | 优先级 | 映射 SYS | 映射 MOD | 映射 IF |
|----|---------|--------|---------|---------|--------|
| **FR-012** | 系统应在 Review 通过后自动触发 pytest 执行，收集通过/失败数和失败详情 | P0 | SYS-005 | MOD-008 | IF-009 |
| **FR-013** | 系统应支持配置通过阈值（默认 4.0/5.0），低于阈值视为测试失败 | P1 | SYS-005 | MOD-008 | IF-009 |

**验收标准**：pytest 返回 3 passed / 1 failed 时，`TestResult.passed_count == 3, failed_count == 1`。

### 2.6 状态机管理

| ID | 需求描述 | 优先级 | 映射 SYS | 映射 MOD | 映射 IF |
|----|---------|--------|---------|---------|--------|
| **FR-014** | 系统应实现 11 状态的任务状态机（CREATED → PASSED/ESCALATED），校验所有状态转换的合法性 | P0 | SYS-006 | MOD-009 | IF-010 |
| **FR-015** | 非法状态转换应抛出 StateMachineError，不允许静默跳过 | P0 | SYS-006 | MOD-009 | IF-010 |

**验收标准**：从 CREATED 直接转 TESTING → 抛出异常；CREATED → QUEUED → DISPATCHED → CODING_DONE → REVIEW → TESTING → JUDGING → PASSED 全链路无异常。

### 2.7 通知与报告

| ID | 需求描述 | 优先级 | 映射 SYS | 映射 MOD | 映射 IF |
|----|---------|--------|---------|---------|--------|
| **FR-016** | 系统应通过钉钉 Webhook 发送 Markdown 格式的实时通知（任务完成/失败/升级） | P0 | SYS-007 | MOD-010 | IF-010 |
| **FR-017** | 系统应在每日 18:00 自动生成每日开发摘要报告（任务统计+代码变更+测试趋势） | P1 | SYS-007 | MOD-010 | IF-010 |
| **FR-018** | 系统应在 Sprint 结束时生成 Sprint 报告（里程碑+全量统计+遗留问题） | P1 | SYS-007 | MOD-010 | IF-010 |

**验收标准**：`reporter.daily_summary()` 生成包含任务统计表格的 Markdown 并发送到钉钉。

### 2.8 Git 自动化

## §4 约束与假设

> 每项使用 **CON-xxx** 编号。

### 4.1 技术约束

| ID | 约束描述 | 影响 |
|----|---------|------|
| **CON-001** | 编码引擎固定为 aider CLI（不自行实现编码能力） | Orchestrator 只做调度，不重复 aider 已有能力 |
| **CON-002** | LLM 通过 OpenAI 兼容 API 调用（一步 API 网关） | 所有 LLM 调用统一走 `OPENAI_API_BASE` + `OPENAI_API_KEY` |
| **CON-003** | Python ≥ 3.10，核心依赖：PyYAML, asyncssh, httpx, loguru | 最小化第三方依赖 |
| **CON-004** | 配置文件格式为 YAML，环境变量用 `${VAR}` 语法引用 | config.py 统一展开 |

### 4.2 资源约束

| ID | 约束描述 | 影响 |
|----|---------|------|
| **CON-005** | 初始机器池为 4 台异构机器（GPU/SSD/通用） | 并行度上限 = 机器数 |
| **CON-006** | 所有机器位于同一内网段（172.16.x.x），SSH 互通 | 无公网暴露需求 |
| **CON-007** | Orchestrator 以单实例运行（无高可用集群） | systemd + watchdog 保障可用性 |

### 4.3 前置假设

| ID | 假设描述 | 风险等级 |
|----|---------|---------|
| **CON-008** | 每台开发机已预装 aider CLI 并配置好 SSH 密钥 | 低 — 已验证部署脚本 |
| **CON-009** | 项目文档集由人工提前编写好，格式符合标准 | 中 — 需文档模板引导 |
| **CON-010** | 内网带宽充足，SSH 传输延迟 < 10ms | 低 — 已测量 |
| **CON-011** | LLM API 可用率 ≥ 99%，响应时间 < 30s | 中 — 依赖第三方服务 |
