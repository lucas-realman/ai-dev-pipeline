## 一、部署架构评审

### 1.1 集群拓扑

ARCH-002 定义 5 台物理机作为初始环境:

| 节点 | IP | 角色 | 评价 |
|------|-----|------|------|
| orchestrator | 172.16.14.201 | 调度中心 | 🔴 **单点故障** — 无 HA 方案 |
| 4090 | 172.16.11.194 | GPU worker | ✅ 合理 |
| mac_mini_8T | 172.16.12.50 | Apple 编码节点 | ✅ 合理 |
| gateway | 172.16.14.215 | 网关/路由 | ⚠️ 职责不明确 |
| data_center | 172.16.14.90 | 数据中心 | ⚠️ 存储策略未说明 |

### 1.2 🔴 单点故障分析

| 单点 | 影响 | 当前缓解 | 建议 |
|------|------|---------|------|
| Orchestrator 主机宕机 | **全系统停摆** | 无 | 部署 systemd watchdog, 数据持久化 |
| LLM API 不可用 | 所有分析/审查停止 | RISK-001 降级 | 需验证降级逻辑已实现 |
| SSH 连接中断 | worker 不可达 | RISK-002 重试 | 需定义重试上限和熔断 |
| 内存任务队列丢失 | **所有进行中任务永久丢失** | 无 | 🔴 必须做状态持久化 |

### 1.3 🔴 状态持久化缺失

这是生产环境部署的**最大风险**:

- `task_engine` 所有任务保存在 Python dict 中 (内存)
- `state_machine` 状态转换历史无持久化
- `machine_registry` 机器列表无持久化

**意味着**: Orchestrator 进程重启 = 所有任务状态归零

**建议**:
1. 最小方案: 使用 SQLite + WAL 模式 (零部署依赖)
2. 推荐方案: Redis + AOF (高性能，支持探活)
3. 在 Sprint 0 完成持久化层设计，Sprint 1 实现

### 2.2 日志策略建议

```
# 建议日志规范
{
    "timestamp": "ISO-8601",
    "level": "INFO|WARN|ERROR",
```

应配合 logrotate 或 systemd journal 进行日志轮转，保留 30 天。

## 三、安全与合规

### 3.1 安全评审

| 检查项 | 状态 | 说明 |
|--------|------|------|
| SSH 密钥管理 | ⚠️ | ARCH-002 提到密钥认证，但无密钥轮换策略 |
| LLM API Key | 🔴 | 配置文件明文存储，无加密方案 |
| 网络隔离 | ⚠️ | 172.16.x.x 内网，但无防火墙规则说明 |
| 代码注入风险 | 🔴 | AI 生成的代码直接在 worker 上执行，无沙箱 |
| 审计日志 | 🔴 | 无操作审计（谁触发了什么任务） |

### 3.2 🔴 AI 生成代码执行安全

系统核心功能是让 AI 生成代码并自动执行。这带来严重安全风险:
- `remote_executor` 通过 SSH 在远程机器执行 AI 生成的命令
- 无沙箱隔离 (非容器化)
- 无命令白名单/黑名单
- 无文件系统只读保护

**建议**:
1. 短期: 限定工作目录 (`chroot` 或 `cwd` 限制)
2. 中期: Docker 容器执行 (SYS-001 ADR-005 已规划但在 v2.0)
3. 长期: gVisor/Kata Containers 安全沙箱

## 四、SLA/SLO 定义

OPS-003 和整体文档缺少生产级 SLA 定义:

| 建议 SLO | 指标 | 目标值 |
|----------|------|-------|
| 可用性 | Orchestrator uptime | ≥ 99.5% (月度) |
| 任务延迟 | 从提交到开始执行 | ≤ 30 秒 |
| 任务完成率 | 成功/总数 | ≥ 85% (含 AI 不确定性) |
| 事件响应 | P0 告警到介入 | ≤ 15 分钟 |
| 恢复时间 | 节点故障到恢复 | ≤ 30 分钟 |

## 五、容量规划

### 5.1 当前容量

| 资源 | 当前值 | 瓶颈预估 |
|------|--------|---------|
| Worker 数量 | 4 台 | 并行 4 任务，但 GPU 节点仅 1 台 |
| 内存限制 | 未定义 | 全内存任务队列，任务数 >1000 可能 OOM |
| 磁盘空间 | 未定义 | Git 仓库 + 日志累积风险 |
| LLM API 并发 | 未限制 | 无令牌桶/限流，可能触发 429 |

### 5.2 扩展建议

1. **Sprint 0**: 定义每台机器的资源限额 (CPU/MEM/Disk)
2. **Sprint 1**: 实现 LLM API 限流 (令牌桶)
3. **Sprint 2**: 监控指标采集 (systemd + cron 脚本最小方案)
4. **v2.0**: 容器化后使用 K8s 资源配额
