## 二、发现的架构级问题

### 2.1 🔴 严重: 缺少状态持久化设计

当前 TaskEngine 的任务队列和 StateMachine 的状态全部存储在 **内存** 中。NFR-005 提到 "Orchestrator 崩溃后自动恢复"，但 systemd 重启后内存数据全丢。

**影响**: 恢复后所有进行中的任务状态丢失，需要人工重新启动 Sprint。

**建议**:
1. 短期: 每次状态变更时写入 `state.json` 文件 (单文件 + fsync)
2. 中期: 使用 SQLite 作为本地任务数据库
3. 长期: Redis + 事件溯源 (Event Sourcing)

### 2.2 🟡 中等: 缺少 LLM 调用抽象层

当前 `doc_analyzer.py` 和 `reviewer.py` 各自直接调用 httpx 访问 LLM API。如果未来切换 LLM 提供商或增加本地推理，需要修改多处。

**建议**: 抽取 `LLMClient` 工具类，封装 `chat_completion(messages, model, temperature)` 接口，所有 LLM 调用走统一入口。

### 2.3 🟡 中等: 缺少健康检查和心跳机制

ARCH-002 提到 5 台机器拓扑，但缺少:
- Worker 心跳上报机制
- Orchestrator 主动探活 (SSH ping)
- 机器状态自动恢复 (ERROR → ONLINE 条件)

当前仅在分发失败时被动标记 ERROR，缺少主动监控。

### 2.4 🟢 建议: 考虑 Webhook 回调替代轮询

SYS-001 §3.1 描述的 `main.py` 主循环是轮询模式 (poll_interval=10s)。建议长期考虑 Worker 完成后通过 HTTP 回调通知 Orchestrator，减少轮询开销。
