## §1 架构风格与设计原则

> 映射: SYS-001~009, ADR-001~006 → 全局

### 1.1 架构风格

本系统采用 **管道-过滤器 (Pipeline-Filter)** 与 **主控-从属 (Master-Worker)** 混合架构：

| 风格 | 应用点 | 说明 |
|------|--------|------|
| **管道-过滤器** | 任务生命周期 | 文档 → 解析 → 调度 → 编码 → 审查 → 测试 → 判定，每阶段独立可替换 |
| **主控-从属** | 计算分发 | Orchestrator (主控) 通过 SSH 调度 Worker 机器 (从属) |
| **状态机** | 任务管理 | 每个 CodingTask 有 11 个状态、严格转换规则 |
| **分层** | 代码组织 | L0 接入 → L1 调度 → L2 执行 → L3 基础设施 (见 SYS-001 §2) |

### 1.2 设计原则

| # | 原则 | 具体体现 | 映射 |
|---|------|---------|------|
| P1 | **薄调度层** | 不重复 aider 已有能力；只做跨机器调度 + 编排 + 汇总 + 通知 | ADR-001 |
| P2 | **文档驱动** | 所有任务从需求文档集自动生成，不接受人工命令式输入 | ADR-002 |
| P3 | **单一职责** | 每个模块一个 `.py` 文件，一个核心类，<300 行 | MOD-001~013 |
| P4 | **可替换执行器** | dispatcher/reviewer/test_runner 接口稳定，实现可替换 | ARCH-003~005 |
| P5 | **失败安全** | 任何单点故障不阻塞整条流水线；机器掉线自动迁移任务 | NFR-004~006 |
| P6 | **内网封闭** | 系统运行在 172.16.x.x 内网，唯一出站为云端 LLM API | CON-002 |
| P7 | **零手动操作** | 正常路径 0 人工；仅异常升级时钉钉通知 | FR-016~018 |

### 1.3 质量属性场景

| 属性 | 刺激 | 响应 | 度量 |
|------|------|------|------|
| **性能** | 4 台机器同时编码 | 不阻塞调度循环 | 调度延迟 < 2s |
| **可靠** | Worker 机器 SSH 断连 | 标记 offline，任务迁移至其他 idle 机器 | 自动恢复 < 30s |
| **可扩展** | 新增第 6 台机器 | 在 config.yaml 添加条目或 API 注册 | 0 代码变更 |
| **安全** | LLM API 密钥泄露 | 密钥仅存储在 .env，不进入 Git | 密钥管理合规 |
| **可观测** | 任务执行异常 | 钉钉实时通知 + 日志文件 | 感知延迟 < 1min |

**关键异常规则**:

| 异常场景 | 状态转换 | 最大重试 | 兜底 |
|----------|---------|---------|------|
| Review 失败 | REVIEW → RETRY → DISPATCHED | review_retry ≤ 3 | ESCALATED → 钉钉通知人工 |
| 测试失败 | FAILED → RETRY → DISPATCHED | test_retry ≤ 3 | ESCALATED → 钉钉通知人工 |
| SSH 超时 | DISPATCHED → RETRY | 1 次自动迁移 | ESCALATED |
| 机器离线 | machine.status → OFFLINE | 任务自动迁移 | 无空闲机器时排队等待 |

## §5 健康检查与心跳机制

> 映射: A-010 (技术总监 + 生产负责人评审要求), NFR-004~006

### 5.1 健康检查架构

```
Orchestrator (主控)
│
├── [定时器] 每 60s 执行一次 health_check_all()
│   │
│   ├── → SSH ping 每台 worker 机器 (timeout 5s)
│   │   ├── 成功 → status = ONLINE, 更新 last_heartbeat
│   │   └── 失败 → retry 2 次 → 仍失败 → status = ERROR
│   │
│   ├── → 检查 LLM API 可达性 (HEAD /v1/models, timeout 10s)
│   │   └── 失败 → 钉钉告警 "LLM API 不可达"
│   │
│   └── → 检查磁盘空间 (.autodev/state.db 所在分区)
│       └── < 500MB → 钉钉告警 "磁盘空间不足"
│
├── [事件驱动] dispatch_task 超时
│   └── → 标记机器 ERROR + 任务回收到 RETRY
│
└── [watchdog] systemd 看门狗
    └── → 进程异常退出 → 自动重启 → load_tasks() 恢复
```

### 5.2 心跳协议

| 属性 | 值 |
|------|---|
| **检测方式** | Orchestrator → Worker SSH 执行 `echo ok` (主动探测) |
| **探测间隔** | 60 秒 (可配置 `health_check_interval`) |
| **超时阈值** | 5 秒 / 次 |
| **重试次数** | 2 次 (间隔 5s) |
| **失败动作** | 标记 `machine.status = ERROR`, 该机器上的 DISPATCHED 任务迁移至其他 idle 机器 |
| **恢复检测** | 下次心跳成功 → 自动恢复 `machine.status = ONLINE` |

### 5.3 影响模块

| 模块 | 变更 |
|------|------|
| **MachineRegistry (MOD-003)** | 新增 `last_heartbeat: float` 字段, `health_check(machine) → bool` 方法 |
| **Orchestrator (MOD-013)** | 主循环增加定时 `health_check_all()` 调度 |
| **Config (MOD-012)** | 新增 `health_check_interval`, `health_check_timeout` 配置项 |
| **Reporter (MOD-010)** | 新增 `notify_machine_error(machine_id)` 告警 |
