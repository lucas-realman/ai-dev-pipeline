## §2 架构决策逐项评审

### ADR-001: SSH 命令行执行 vs Agent 模式

**当前方案**: 通过 `ssh` / `scp` 系统命令远程执行 aider  
**替代方案**: 在每台机器部署 Agent 守护进程 + gRPC/HTTP API

| 对比维度 | SSH 命令行 (当前) | Agent 模式 (替代) |
|----------|------------------|-------------------|
| 部署复杂度 | ✅ 零额外部署 — 只需 SSH 密钥 | ❌ 需在每台机器安装/维护 Agent |
| 实时通信 | ❌ 单次执行，无中间状态 | ✅ 双向流式通信 |
| 监控能力 | ⚠️ 只能通过 SSH 心跳检测 | ✅ 实时 CPU/内存/任务进度 |
| 安全性 | ✅ SSH 密钥认证，业界标准 | ⚠️ 需自行实现认证和TLS |
| 故障恢复 | ⚠️ 进程中断无法续传 | ✅ Agent 可自动恢复 |
| 横向扩展 | ⚠️ SSH 连接数有实际限制 (~100) | ✅ 连接池 + 负载均衡 |

**评审意见**:
- **当前阶段合理** — 4 台机器规模下 SSH 命令行是最优选择，零运维成本
- **演进路径明确** — 当机器池 > 20 台时，建议引入轻量 Agent (如 gRPC)
- 当前代码中 `Dispatcher._ssh_exec()` 已通过 `asyncio.create_subprocess_exec` 封装，替换为 Agent 通信时改动集中在此方法

**结论**: ✅ 合理 — 近期 SSH 足够，远期演进路径清晰

### ADR-002: asyncio 并发模型 vs 多线程/多进程

**当前方案**: asyncio 协程处理 I/O 密集操作 (SSH, HTTP, LLM 调用)  
**辅助**: threading.Lock 保护共享状态 (MachineRegistry, TaskEngine)

| 对比维度 | asyncio (当前) | 多线程 | 多进程 |
|----------|---------------|--------|--------|
| I/O 并发 | ✅ 原生支持 | ⚠️ GIL 下可行 | ❌ IPC 开销 |
| CPU 密集 | ❌ 不适合 | ⚠️ GIL 限制 | ✅ 无 GIL |
| 状态共享 | ✅ 单线程共享 | ⚠️ 需要锁 | ❌ 需 IPC |
| 复杂度 | ⚠️ async/await 学习曲线 | ✅ 简单直觉 | ❌ 最复杂 |
| SSH 执行 | ✅ subprocess → asyncio | ✅ subprocess | ✅ subprocess |

**评审意见**:
- **完全合理** — 核心操作 (SSH执行、LLM调用、钉钉通知) 全部是 I/O 密集
- asyncio + `create_subprocess_exec` 执行 SSH 是正确选择
- `threading.Lock` 在 `MachineRegistry` 和 `TaskEngine` 中保护状态 — 混用 asyncio+threading 需要注意:
  - ✅ Lock 操作都很短 (dict 读写)，不会阻塞 event loop
  - ⚠️ 但如果未来在异步上下文中持锁时间变长，应迁移到 `asyncio.Lock`
- 无 CPU 密集计算 — 无需多进程

**潜在问题**:
- `dispatch_batch()` 使用 `asyncio.gather()` 并行 SSH — 如果 4 台机器同时超时 (600s)，会占用 event loop 达 10 分钟
- 建议: `asyncio.gather(*coros, return_exceptions=True)` 隔离单任务异常

**结论**: ✅ 合理

### ADR-003: LLM 降级通过策略

**当前方案**: LLM API 调用失败时，Reviewer L2/L3 降级为"通过" (score=4.0)  
**替代方案**: A) 重试 3 次后失败; B) 排队等待 API 恢复; C) 本地模型回退

| 对比维度 | 降级通过 (当前) | 重试+失败 | 排队等待 | 本地模型回退 |
|----------|---------------|----------|---------|-------------|
| 流水线连续性 | ✅ 永不中断 | ❌ 可能阻塞 | ❌ 可能长时间等待 | ✅ 不中断 |
| 代码质量保障 | ⚠️ 跳过 LLM 审查 | ✅ 确保审查 | ✅ 确保审查 | ⚠️ 质量降低 |
| 实现复杂度 | ✅ 最简单 | ✅ 简单 | ⚠️ 需队列管理 | ❌ 需部署本地模型 |
| 运维成本 | ✅ 无 | ✅ 无 | ⚠️ 监控队列 | ❌ GPU 资源 |

**评审意见**:
- **当前阶段合理** — 优先保证编码流水线不中断，事后通过测试兜底
- **score=4.0 风险**: `pass_threshold` 默认也是 4.0，降级分数刚好在阈值线上
  - 如果 score=4.0 不触发 RETRY 而是通过，等于审查完全跳过
  - **建议**: 降级 score 设为 `pass_threshold - 0.5` (即 3.5)，触发 RETRY 但不 ESCALATE
- **演进建议**: 增加 LLM 重试 (1-2 次, 指数退避)，减少降级频率

**结论**: ⚠️ 基本合理但 score=4.0 存在阈值边界问题

### ADR-004: 内存状态管理 vs 数据库持久化

**当前方案**: `OrderedDict` 内存存储所有任务状态，无持久化  
**替代方案**: A) SQLite 嵌入式; B) Redis; C) JSON 文件快照; D) PostgreSQL

| 对比维度 | OrderedDict (当前) | SQLite | Redis | JSON 快照 | PostgreSQL |
|----------|-------------------|--------|-------|----------|------------|
| 性能 | ✅ 最快 | ✅ 快 | ✅ 快 | ⚠️ 写盘慢 | ⚠️ 网络延迟 |
| 持久化 | ❌ 无 | ✅ 完整 | ⚠️ AOF | ✅ 每快照 | ✅ 完整 |
| 部署复杂度 | ✅ 零依赖 | ✅ 内置 | ❌ 需部署 | ✅ 零依赖 | ❌ 需部署 |
| 查询能力 | ⚠️ 遍历 | ✅ SQL | ✅ 键值 | ❌ 全量加载 | ✅ SQL |
| 并发安全 | ⚠️ 需要锁 | ✅ 事务 | ✅ 原子操作 | ⚠️ 需要锁 | ✅ 事务 |

**评审意见**:
- **短期可接受** — Sprint 执行时间 < 1 小时，进程崩溃概率低
- **中期必须改进** — DD-MOD-006 已设计 JSON 快照持久化但未实现
- **推荐路径**: OrderedDict → JSON 快照 (v1.1) → SQLite (v2.0)
  - JSON 快照: 每次状态变更写入 `state_snapshot.json`，重启时加载
  - SQLite: 当任务数 > 100 或需历史查询时迁移

**风险量化**:
- 单次 Sprint 通常 4-12 个任务，执行时间 15-60 分钟
- Python 进程在此期间崩溃概率约 < 0.1%
- 但如果崩溃，所有已完成任务需重新执行 — **浪费时间 = 已执行时间 × 任务完成比例**

**结论**: ⚠️ 有条件合理 — 必须在 v1.1 实现 JSON 快照持久化

### ADR-005: aider 作为唯一编码引擎

**当前方案**: 只支持 aider CLI 作为编码执行器  
**替代方案**: A) 可插拔引擎 (aider / cursor / codestory); B) 自建 LLM → 代码生成

| 对比维度 | aider (当前) | 可插拔引擎 | 自建代码生成 |
|----------|------------|-----------|-------------|
| 实现成本 | ✅ 零 — 直接调用 | ⚠️ 需抽象层 | ❌ 极高 |
| 编码质量 | ✅ aider 成熟 | ✅ 可选最优 | ⚠️ 需大量调优 |
| 灵活性 | ⚠️ 锁定 aider | ✅ 可替换 | ✅ 完全控制 |
| 维护成本 | ✅ aider 社区维护 | ⚠️ 多引擎适配 | ❌ 自行维护 |

**评审意见**:
- **合理** — aider 是目前最成熟的 CLI AI 编码工具，直接使用是务实选择
- 编码引擎替换只需修改 `_build_ssh_script()` 中的 aider 命令行 — **改动集中，低耦合**
- 建议在 v2.0 引入 `CodingEngine` 抽象接口:
  ```python
  class CodingEngine(Protocol):
      async def execute(self, task: CodingTask, machine: MachineInfo) -> TaskResult: ...
  ```

**结论**: ✅ 合理

## §4 可扩展性分析

### 4.1 水平扩展路径

```
当前 (v1.0): 4 台机器, 单 Orchestrator
  ↓
v1.1: 10-20 台机器, 单 Orchestrator + JSON 快照
  ↓
v2.0: 20-50 台机器, 单 Orchestrator + SQLite + Agent 模式
  ↓
v3.0: 50+ 台机器, Orchestrator 集群 + 消息队列 + 数据库
```

### 4.2 当前架构瓶颈

| 瓶颈 | 触发条件 | 影响 | 缓解方案 |
|------|---------|------|---------|
| SSH 连接数 | > 50 台机器并发 | 连接建立延迟增加 | 连接池 / Agent 模式 |
| 内存占用 | > 1000 个任务 | OrderedDict 增长 | SQLite 持久化 |
| LLM API 限流 | > 10 req/min | 429 错误增加 | 请求队列 + 退避 |
| 单点故障 | Orchestrator 崩溃 | 全部状态丢失 | 状态持久化 + 热备 |
| Git 推送冲突 | > 8 台机器同时推送 | 重试增加，延迟增大 | git bare repo + merge queue |

### 4.3 扩展改动量估算

| 扩展项 | 影响模块 | 改动量 | 风险 |
|--------|---------|--------|------|
| 增加新机器 | config.yaml + MachineRegistry | ~10 行配置 | 极低 |
| 替换编码引擎 | dispatcher._build_ssh_script() | ~50 行代码 | 低 |
| 添加新通知渠道 | reporter.py | ~80 行代码 | 低 |
| 状态持久化 | task_engine.py | ~150 行代码 | 中 |
| Agent 模式替换 SSH | dispatcher.py | ~200 行代码 | 中 |
| Orchestrator 集群 | 全部模块 | ~1000 行代码 | 高 |
