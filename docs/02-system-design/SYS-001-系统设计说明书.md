### 1.2 系统边界定义

| 边界 | 系统内 | 系统外 |
|------|--------|--------|
| **编码** | 调度 aider、收集结果 | aider 本身（第三方 CLI） |
| **LLM** | 构造 prompt、解析 JSON 响应 | LLM 推理服务 |
| **Git** | pull/push/tag 自动化 | Git 服务器/裸仓库 |
| **通知** | 构造 Markdown、发送 HTTP | 钉钉服务端 |
| **机器** | 注册/调度/监控 | 物理机器维护 |

### 1.3 Actor 清单

| Actor | 角色 | 与系统的交互 |
|-------|------|-------------|
| **开发者** | 系统唯一人类用户 | 编写文档集 → push；接收钉钉通知 → 决策 (继续/暂停) |
| **aider CLI** | AI 编码执行器 | 被 Dispatcher SSH 调用，执行编码并 auto-commit |
| **LLM API** | 智能推理服务 | DocAnalyzer 调用 → 任务拆解；Reviewer L2/L3 调用 → 审查 |
| **钉钉 API** | 消息推送通道 | Reporter 发送 Markdown 通知 |
| **Git 裸仓库** | 版本管理 | push/pull/hook 触发 |

## §2 分层架构 (SYS-002, SYS-003)

> 映射: FR-004~008, NFR-001~003, NFR-007 → ARCH-001, ARCH-002, MOD-003/004/006

### 2.1 四层模型

```
┌─────────────────────────────────────────────────────────────────────┐
│  L0  接入层 (Entry)                                                │
│  ┌───────────┐  ┌────────────┐  ┌──────────────┐  ┌────────────┐  │
│  │ CLI 入口  │  │ Git Hook   │  │ 文档变更监听 │  │ 定时触发器 │  │
│  │ (autodev) │  │ post-recv  │  │  (watchdog)  │  │  (cron)    │  │
│  └─────┬─────┘  └─────┬──────┘  └──────┬───────┘  └─────┬──────┘  │
│        └───────────────┴───────────────┴─────────────────┘         │
│                                  │                                  │
├──────────────────────────────────┼──────────────────────────────────┤
│  L1  调度层 (Orchestration)      ▼                                  │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐              │
│  │ doc_analyzer │  │ task_engine  │  │ state_machine│              │
│  │ (文档解析)   │→│ (任务引擎)  │→│ (状态管理)  │              │
│  └──────────────┘  └──────┬───────┘  └──────────────┘              │
│                           │                                         │
├───────────────────────────┼─────────────────────────────────────────┤
│  L2  执行层 (Execution)   ▼                                         │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐              │
│  │ dispatcher   │  │ reviewer     │  │ test_runner  │              │
│  │ (SSH 分发)   │  │ (三层审查)   │  │ (测试执行)   │              │
│  └──────────────┘  └──────────────┘  └──────────────┘              │
│                                                                     │
├─────────────────────────────────────────────────────────────────────┤
│  L3  基础设施层 (Infrastructure)                                     │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐              │
│  │ machine_     │  │ git_ops      │  │ reporter     │              │
│  │  registry    │  │ (Git 操作)   │  │ (通知/报告)  │              │
│  │ (机器池)     │  │              │  │              │              │
│  └──────────────┘  └──────────────┘  └──────────────┘              │
│                                                                     │
│  ┌──────────────┐  ┌──────────────┐                                │
│  │ config       │  │ task_models  │                                │
│  │ (配置)       │  │ (数据模型)   │                                │
│  └──────────────┘  └──────────────┘                                │
└─────────────────────────────────────────────────────────────────────┘
```

| 层 | 职责 | 模块 | 对应 SYS |
|----|------|------|---------|
| **L0 接入层** | 接收外部触发，启动流水线 | CLI (main.py)、Git Hook、watchdog | SYS-001 |
| **L1 调度层** | 文档解析→任务拆解→状态流转 | doc_analyzer, task_engine, state_machine | SYS-003, SYS-006 |
| **L2 执行层** | SSH 编码、自动审查、自动测试 | dispatcher, reviewer, test_runner | SYS-003~005 |
| **L3 基础设施层** | 机器池、Git、通知、配置、数据模型 | machine_registry, git_ops, reporter, config, task_models | SYS-002, SYS-007~009 |

### 2.2 层间依赖规则

| 规则 | 说明 |
|------|------|
| **单向依赖** | L0 → L1 → L2 → L3；禁止反向调用 |
| **跨层禁止** | L0 不得直接调用 L2/L3 模块 |
| **L3 无状态** | 基础设施层模块仅提供原子操作，不持有业务状态 |
| **L1 持有状态** | TaskEngine + StateMachine 是唯一状态持有者 |
| **L2 可替换** | 执行层模块可独立替换（如 dispatcher 从 SSH 切换到 K8s Job） |

## §3 通信模型 (SYS-003, SYS-007, SYS-008)

> 映射: FR-006~008, FR-016~020, NFR-001~003, NFR-010, NFR-013 → ARCH-001, MOD-006/010/011

### 3.1 内部通信 (SYS-003)

系统内部模块通过 **同进程函数调用** 交互，不引入 RPC/消息队列（除未来可选 Redis 任务队列外）。

```
┌────────────────────────────────────────────────────────────────┐
│  main.py (主循环 / FastAPI)                                    │
│                                                                │
│  1. doc_analyzer.analyze_and_decompose(sprint_id)              │
│         │                                                      │
│         ▼ List[CodingTask]                                     │
│  2. task_engine.enqueue(tasks)                                 │
│         │                                                      │
│         ▼ List[CodingTask]                                     │
│  3. task_engine.next_batch(machine_registry)                   │
│         │    ↕ machine_registry.get_idle_machines()             │
│         │    ↕ machine_registry.match_machine(tags)             │
│         ▼ List[CodingTask]                                     │
│  4. dispatcher.dispatch_batch(tasks)                           │
│         │                                                      │
│         ▼ List[TaskResult]                                     │
│  5. reviewer.review(result)                                    │
│         │                                                      │
│         ▼ ReviewResult                                         │
│  6. state_machine.transition(task, event)                      │
│         │ ↕ task_engine.handle_result()                         │
│         ▼                                                      │
│  7. reporter.notify(summary)                                   │
│         │                                                      │
│         ▼ HTTP POST (钉钉)                                     │
└────────────────────────────────────────────────────────────────┘
```

**数据流关键对象**：

| 数据对象 | 产生者 | 消费者 | 序列化 |
|----------|--------|--------|--------|
| `CodingTask` | doc_analyzer | task_engine, dispatcher | dataclass → JSON |
| `TaskResult` | dispatcher | reviewer, state_machine | dataclass |
| `ReviewResult` | reviewer | state_machine, reporter | dataclass |
| `TestResult` | test_runner | state_machine, reporter | dataclass |
| `MachineInfo` | machine_registry | task_engine, dispatcher | dataclass → YAML |

### 3.2 外部通信 (SYS-007, SYS-008)

| 通道 | 协议 | 方向 | 端点 | 认证 | 超时 |
|------|------|------|------|------|------|
| **SSH 编码** | SSH (asyncssh) | Orchestrator → Worker | `{host}:{port}` | 密钥 (~/.ssh) | 600s/任务 |
| **LLM API** | HTTPS (OpenAI 兼容) | Orchestrator → 一步API网关 | `http://120.133.40.59/api/v1` | Bearer Token | 120s |
| **钉钉通知** | HTTPS POST | Orchestrator → 钉钉 | Webhook URL | URL 内含签名 | 10s |
| **Git 推送** | SSH/HTTPS | Worker → Git 裸仓库 | `mac_min_8T` bare repo | SSH 密钥 | 60s |
| **Git Hook** | Shell → HTTP | Git 裸仓库 → Orchestrator | post-receive → curl | 内网无认证 | 5s |

**外部通信时序**（单任务完整生命周期）：

```
Orchestrator      Worker机器      LLM API       Git裸仓库      钉钉
    │                │               │              │           │
    │──SSH connect──▶│               │              │           │
    │──git pull─────▶│               │              │           │
    │──aider cmd────▶│               │              │           │
    │                │──LLM请求─────▶│              │           │
    │                │◀─LLM响应──────│              │           │
    │                │──auto-commit─▶│──git push──▶│           │
    │◀─TaskResult────│               │              │           │
    │                                │              │           │
    │──Review请求───────────────────▶│              │           │
    │◀─ReviewResult─────────────────│              │           │
    │                                               │           │
    │──git tag+push─────────────────────────────── ▶│           │
    │                                               │           │
    │──通知────────────────────────────────────────────────────▶│
    │                                                           │
```

## §4 关键设计决策 (ADR)

> 映射: 全局 → ARCH-001, ARCH-002

### 4.1 决策记录 (Architecture Decision Records)

#### ADR-001: 薄调度层模式

| 项目 | 内容 |
|------|------|
| **决策** | Orchestrator 定位为"薄调度层"，不重复实现 aider 已有能力 |
| **上下文** | aider 已覆盖：上下文读取 → LLM 代码生成 → 文件写入 → auto-commit → lint/test → 失败重试 |
| **推论** | Orchestrator 只做 aider 做不到的事：跨机器调度 + 任务编排 + 结果汇总 + 通知 |
| **映射** | FR-006, FR-007, FR-008 → SYS-003 |
| **状态** | ✅ 已采纳 |

#### ADR-002: 文档驱动任务拆解

| 项目 | 内容 |
|------|------|
| **决策** | 用 LLM 读取整套需求文档集，AI 理解全貌后自动拆解为编码任务 |
| **备选** | (a) 人工编写任务卡 (b) 只解析 Sprint 任务卡表格 (v2.0 方式) |
| **理由** | 完整文档集提供需求+设计+契约+测试的端到端上下文，AI 拆解粒度可控 (5-15 min/task) |
| **映射** | FR-001~003 → SYS-001 |
| **状态** | ✅ 已采纳 |

#### ADR-003: 动态机器池 vs 硬编码

| 项目 | 内容 |
|------|------|
| **决策** | Machine Registry 提供动态注册/注销/标签匹配，不硬编码机器 |
| **备选** | (a) config.yaml 写死 5 台 (v2.0) (b) Kubernetes 自动扩缩 |
| **理由** | 当前 5 台物理机，K8s 过重；动态注册兼容未来扩展 |
| **约束** | 所有机器须在 172.16.x.x 内网，SSH 密钥预配置 |
| **映射** | FR-004~005, CON-002 → SYS-002 |
| **状态** | ✅ 已采纳 |

#### ADR-004: 云端 API vs 本地 vLLM 分工

| 项目 | 内容 |
|------|------|
| **决策** | 编码/审查/文档解析走云端 Claude Opus 4.6 (一步API)；Agent 推理走本地 vLLM (Qwen3-30B) |
| **理由** | 编码/审查需要强模型能力；Agent 推理走内网 vLLM 零延迟、与生产配置一致 |
| **风险** | 云端 API 限流 → 配置速率限制 + 重试 |
| **映射** | FR-009, CON-003 → SYS-004 |
| **状态** | ✅ 已采纳 |

#### ADR-005: 同步主循环 vs 事件驱动

| 项目 | 内容 |
|------|------|
| **决策** | 采用 async 主循环 + asyncssh 并发，不引入消息队列 |
| **备选** | (a) Redis pub/sub (b) Celery worker (c) 纯多线程 |
| **理由** | 当前并发上限 4 台机器，async 足够；避免引入分布式中间件复杂度 |
| **演进** | 机器池 >10 台时考虑 Redis 任务队列 |
| **映射** | FR-006~008, NFR-001~003 → SYS-003 |
| **状态** | ✅ 已采纳 |

#### ADR-006: 三层 Review 架构

| 项目 | 内容 |
|------|------|
| **决策** | L1 静态检查 (lint/mypy) → L2 LLM 语义审查 → L3 三模型交叉评估 |
| **理由** | L1 成本为零，过滤低级错误；L2 检查逻辑/契约；L3 防止单模型偏差 |
| **触发** | 每个任务 CODING_DONE 后自动进入 REVIEW |
| **映射** | FR-009~011 → SYS-004 |
| **状态** | ✅ 已采纳 |

#### ADR-007: 状态持久化方案 (SQLite)

| 项目 | 内容 |
|------|------|
| **决策** | 采用 SQLite 作为任务状态及机器状态的持久化存储 |
| **上下文** | 当前 TaskEngine 使用纯内存 `List[CodingTask]` + `Dict` 管理任务队列，进程崩溃或重启后全部状态丢失，无法恢复 Sprint 执行进度。这是评审共识 #2 (技术总监/生产负责人/CEO) 指出的 P0 阻塞问题。 |
| **备选方案** | (a) SQLite 嵌入式 (b) Redis (c) PostgreSQL (d) JSON 文件 |
| **选型分析** | |
| | **SQLite**: 零运维、单文件、ACID、Python 内置 `sqlite3`，性能满足 (<50 task/day)。**推荐 ✅** |
| | **Redis**: 需额外部署进程，增加运维负担 (CON-007 单人运维)，适合高并发场景但当前过重 |
| | **PostgreSQL**: 重量级关系型 DB，当前 5 台机器场景无必要 |
| | **JSON 文件**: 无 ACID 保证，并发写入不安全，数据一致性差 |
| **决策理由** | (1) 零额外部署、零运维成本，符合 CON-007 单人约束 (2) Python 内置 `sqlite3`，无新依赖 (3) ACID 事务保证崩溃恢复 (4) 单文件，备份/迁移简单 (5) 性能远超需求 (每天 <50 任务) |
| **存储设计** | |
| | 数据库文件：`{project_path}/.autodev/state.db` |
| | 核心表：`tasks` (CodingTask 全字段), `machines` (MachineInfo 快照), `sprint_log` (Sprint 执行日志) |
| | 索引：`tasks(status)`, `tasks(sprint_id)`, `tasks(assigned_machine)` |
| **接口设计** | |
| | 新增 MOD-014: `persistence.py` —— `StatePersistence` 类 |
| | `save_task(task: CodingTask) → None` — INSERT or UPDATE (upsert) |
| | `load_tasks(sprint_id: int) → List[CodingTask]` — 恢复指定 Sprint 全部任务 |
| | `save_machine_snapshot(machines: List[MachineInfo]) → None` |
| | `load_machine_snapshot() → List[MachineInfo]` |
| | `save_sprint_event(sprint_id, event, detail) → None` — Sprint 审计日志 |
| **迁移方案** | |
| | Phase 1 (Sprint 0): 实现 `StatePersistence`，`TaskEngine` 每次状态变更后调用 `save_task()` |
| | Phase 2 (Sprint 1): `Orchestrator.run_sprint()` 启动时调用 `load_tasks()` 检测是否有未完成任务，支持断点恢复 |
| | Phase 3 (v2.0): 可选切换到 Redis (通过 Protocol 接口抽象 `PersistenceBackend`) |
| **影响模块** | TaskEngine (调用 save), Orchestrator (调用 load), Config (新增 db_path 配置) |
| **映射** | NFR-005 (崩溃恢复) → SYS-006 |
| **状态** | ✅ 已采纳 |
